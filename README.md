# project-management-skill

## 使用說明

這個 repository 包含一套 Domain Knowledge（DK）建立與維護的 skill，用來：
- 引導 Kickoff 訪談並產出 `logic_map/` 下的專案檔案
- 每天在 `log.md` 記錄新發生的事
- 週五用一句話更新 `report.md`

主要檔案：
- 問卷：`dk-system/kickoff_full_questionnaire.md`
- 輸出路徑：`logic_map/{project}_kickoff.md`、`logic_map/{project}_report.md`、`logic_map/{project}_log.md`

常用觸發指令：
- 「我接了一個新專案」→ 顯示 Kickoff 問卷
- 「我剛 Kickoff 完 / 幫我建立邏輯地圖」→ 產出三個檔案
- 「週五到了 / 幫我更新邏輯地圖」→ 更新 report.md，清空 log.md
- 「解釋這段」→ 把技術內容轉為非技術摘要
- 「評估替代方案」→ 產出替代方案含估時和風險
- 「整合舊記錄：[貼上內容]，專案名稱是 XXX」→ 把舊記錄整理成 kickoff.md

## Skill 涵蓋情境清單

詳細步驟參考：[dk-system/references/situations.md](dk-system/references/situations.md)

| 編號 | 情境 | 建議指令 | 預期輸出 / 行為 | 狀態 |
|---|---|---|---|---|
| #01 | 全新專案，你是第一個 TPM | `我接了一個新專案` | 顯示 Kickoff 問卷，填完後產出 `kickoff.md` → 自動建立 `report.md` 和空白 `log.md` | ✅ |
| #02 | 全新專案，PM 答案很模糊 | `這個問題 PM 答不清楚，幫我追問：[貼上 PM 的模糊回答]` | 用舉例法/對比法/量化法追問，把待確認的標注進 report.md | ✅ |
| #03 | 接手有舊記錄 | `整合舊記錄：[貼上舊記錄內容]，專案名稱是 XXX` | 解析舊記錄整理成 `kickoff.md`，再產出 `report.md` 初版，標注「由舊記錄重建，假設需重新驗證」 | ✅ |
| #04 | 前任離職、無紀錄 | `重建 Kickoff` | 發送最小必答問卷，產出最小版 `kickoff.md` 和 onboarding checklist | ✅ |
| #05 | 正常 Refine | `Refine 記要`（貼會議要點） | 把要點整理後記進 `log.md`，週五統一更新 `report.md` | ✅ |
| #06 | 發現假設未驗證 | `標註需驗證` | 在 `log.md` 加入驗證任務，格式：「待驗證：XXX，誰負責，預計時間」 | ✅ |
| #07 | PO 與工程師意見分歧 | `生成討論摘要`（貼會議摘錄） | 產出決策選項比較表（選項/風險/影響），並在 `log.md` 建立待決事項 | ✅ |
| #08 | 技術內容聽不懂 | `解釋這段`（貼摘錄） | 產出非技術摘要 + 2-3 個追問範本，讓你可以在會議中或會後使用 | ✅ |
| #09 | 會議未決事項 | `標註待決事項` | 記進 `log.md`，格式：「待決定：XXX，誰負責，預計回覆時間：XXX」，下次 Refine 前追蹤 | ✅ |
| #10 | 工程師說做不到 | `評估替代方案`（貼上理由） | 澄清「做不到」的意思，產出 2-3 個替代方案含估時和風險 | ✅ |
| #11 | 主管問你（你知道答案） | `快速回覆主管：<問題>` | 從 `report.md` 擷取對應假設，整理為「問題是什麼 + 解法 + 需要主管決定什麼」的格式 | ✅ |
| #12 | 主管問你（邏輯地圖標注待確認） | `列出待確認項` | 直接說「我現在不知道，X 小時內給你答案」，列出待確認條目和建議責任人 | ✅ |
| #13 | 主管問你（邏輯地圖完全沒有） | `補資料：<問題>` | 產出追問清單或跨部門確認的一句話說法 | ✅ |
| #14 | 工程師一直給技術答案 | `轉成決策選項`（貼技術內容） | 轉為 PM 可決策的選項，列出對 timeline/成本/品質的影響 | ✅ |
| #15 | 假設驗證錯誤但 PM 抵抗 | `生成風險與回退計畫` | 產出量化風險表 + 降低風險的最小可行方案，記進 `log.md` 標注「PM 知情後決定繼續推進」 | ✅ |
| #16 | 專案中途大改方向 | `評估變更影響`（貼變更摘要） | 重寫「一句話說清楚」，逐一審查假設（✅/❌/⚠️），產出需要重新 Kickoff 的清單 | ✅ |
| #17 | 你不在時有人做決定 | `同步變更`（貼記錄或 email） | 把變更摘要加入 `report.md`，標注「他人代決」，建議補課順序：工程師→記錄→PO→更新地圖→主管 | ✅ |
| #18 | 同時負責多個專案、記錄混亂 | `抽出專案 {project}` | 依標籤從混亂的 log 抽出特定專案條目，重建該專案的 `log.md` | ⚠️ 待開發 |
| #19 | 一個 bug 影響多專案 | `cross-project bug: <描述>` | 分別打開兩個 report.md，判斷是否同根本原因，產出跨專案影響矩陣 | ⚠️ 待開發 |
| #20 | 假設被推翻，連鎖反應 | `追蹤影響鏈 <假設>` | 往上往下追假設依賴關係，列出受影響項目，跟 PM 說連鎖影響，立刻更新 `report.md` | ✅ |
| #21 | Kickoff 訪談後建立初版 | `建立初版`（貼訪談筆記） | 產出 `kickoff.md`、`report.md` 與空白 `log.md` | ✅ |
| #22 | 週五更新邏輯地圖 | `週五到了` | 把 `log.md` 合入 `report.md`，標注「本週更新」，清空 `log.md`，提醒確認三件事 | ✅ |
| #23 | 新 PM onboarding | `匯出導覽版 {project}` | 用 `report.md` 當 onboarding 文件，產出關鍵待確認項與 30 分鐘議程，趁機重新量化成功定義 | ✅ |
| #24 | 邏輯地圖太大難維護 | `精簡邏輯地圖 {project}` | 按三個時機精簡（功能上線後/假設驗證後/方向大改後），舊內容移到「歷史記錄」不要刪 | ✅ |
| #25 | 資料 schema 頻繁變動 | `schema 異動：<描述>` | 在 `report.md` 核心假設清單標注「格式可能不穩定」，記進 `log.md` 追蹤 | ⚠️ 待開發 |
