# project-management-skill

## 使用說明

這個 repository 包含一套 Domain Knowledge（DK）建立與維護的 skill，用來：
- 引導 Kickoff 訪談並產出 `logic_map/` 下的專案檔案
- 每天在 `log.md` 記錄新發生的事（記得加【專案名稱】標籤）
- 週五用一句話更新 `report.md`

主要檔案：
- 問卷：`dk-system/kickoff_full_questionnaire.md`
- 輸出路徑：`logic_map/{project}_kickoff.md`、`logic_map/{project}_report.md`、`logic_map/{project}_log.md`

常用觸發指令：
- 「我接了一個新專案」→ 顯示 Kickoff 問卷
- 「我剛 Kickoff 完 / 幫我建立邏輯地圖」→ 產出三個檔案
- 「週五到了 / 幫我更新邏輯地圖」→ 更新 report.md，清空 log.md
- 「幫我記進 log：[內容]，專案名稱是 XXX」→ 把每天發生的事記進 log.md
- 「解釋這段」→ 把技術內容轉為非技術摘要
- 「評估替代方案」→ 產出替代方案含估時和風險
- 「整合舊記錄：[貼上內容]，專案名稱是 XXX」→ 把舊記錄整理成 kickoff.md
- 「這個 bug 同時影響 {專案A} 和 {專案B}」→ 對照兩個 report.md，判斷根本原因
- 「幫我準備向主管報告：{專案名稱}，重點是 [什麼]」→ 產出 Bottom line 格式報告
- 「突發狀況：[描述]，專案是 {專案名稱}」→ 走一遍六個問題框架
- 「進度落後：[什麼卡住]，專案是 {專案名稱}」→ 評估連帶影響，更新時程，產出主管 heads-up
- 「依賴卡住：[哪個依賴]，專案是 {專案名稱}」→ 判斷是否進關鍵路徑，建議是否升級
- 「這個需求算不算加太多：[新需求]，專案是 {專案名稱}」→ 對照 Kickoff 邊界，量化影響

**多專案 log 規範：**
```
✅ 正確：「【員工請假排班系統】週三 Refine：決定 Phase 1 只做 web 版」
❌ 錯誤：「週三 Refine：決定 Phase 1 只做 web 版」
```

## Skill 涵蓋情境清單

詳細步驟參考：[dk-system/references/situations.md](dk-system/references/situations.md)

| 編號 | 情境 | 建議指令 | 預期輸出 / 行為 | 狀態 |
|---|---|---|---|---|
| #01 | 全新專案，你是第一個 TPM | `我接了一個新專案` | 顯示 Kickoff 問卷，填完後產出 `kickoff.md` → 自動建立 `report.md` 和空白 `log.md` | ✅ |
| #02 | 全新專案，PM 答案很模糊 | `這個問題 PM 答不清楚，幫我追問：[貼上 PM 的模糊回答]` | 用舉例法/對比法/量化法追問，把待確認的標注進 report.md | ✅ |
| #03 | 接手有舊記錄 | `整合舊記錄：[貼上舊記錄內容]，專案名稱是 {專案名稱}` | 解析舊記錄整理成 `kickoff.md`，再產出 `report.md` 初版，標注「由舊記錄重建，假設需重新驗證」 | ✅ |
| #04 | 前任離職、無紀錄 | `重建 Kickoff` | 先問三個最小問題（專案叫什麼/PM 是誰/最緊急的是什麼），產出最小版 `kickoff.md`，其餘欄位標「待補課」，並列出補課順序 | ✅ |
| #05 | 每天記錄 / 正常 Refine | `幫我記進 log：[貼上今天發生的事或 Refine 重點]，專案名稱是 {專案名稱}` | 把內容加【專案名稱】標籤後記進 `log.md`，週五統一更新 `report.md` | ✅ |
| #06 | 發現假設未驗證 | `標註需驗證：[假設內容]，專案是 {專案名稱}，誰負責驗證` | 在 `log.md` 加入驗證任務（誰負責、預計時間），同時在 `report.md` 核心假設清單標注 ⚠️ 未驗證 | ✅ |
| #07 | PO 與工程師意見分歧 | `生成討論摘要：[PO 說的] vs [工程師說的]，幫我整理決策選項，專案是 {專案名稱}` | 產出決策選項比較表（選項/風險/影響），並在 `log.md` 建立待決事項 | ✅ |
| #08 | 技術內容聽不懂 | `解釋這段`（貼摘錄） | 產出非技術摘要 + 2-3 個追問範本，讓你可以在會議中或會後使用 | ✅ |
| #09 | 會議未決事項 | `標註待決事項：[什麼事情]，由 [誰] 負責，預計 [什麼時間] 回覆，專案是 {專案名稱}` | 記進 `log.md`，格式：「待決定：XXX，誰負責，預計回覆時間：XXX」，下次 Refine 前追蹤 | ✅ |
| #10 | 工程師說做不到 | `評估替代方案：[工程師說的理由]，專案是 {專案名稱}` | 澄清「做不到」的意思，產出 2-3 個替代方案含估時和風險 | ✅ |
| #11 | 主管問你（你知道答案） | `快速回覆主管：[問題內容]，專案是 {專案名稱}` | 從 `report.md` 擷取對應假設，整理為「問題是什麼 + 解法 + 需要主管決定什麼」的格式 | ✅ |
| #12 | 主管問你（邏輯地圖標注待確認） | `我不知道這個答案，幫我看 {專案名稱} 的 report.md，有沒有相關的待確認項` | 直接說「我現在不知道，X 小時內給你答案」，列出待確認條目和建議責任人 | ✅ |
| #13 | 主管問你（邏輯地圖完全沒有） | `補資料：[問題內容]，專案是 {專案名稱}` | 產出追問清單或跨部門確認的一句話說法 | ✅ |
| #14 | 工程師一直給技術答案 | `轉成決策選項：[貼技術內容]，專案是 {專案名稱}` | 轉為 PM 可決策的選項，列出對 timeline/成本/品質的影響 | ✅ |
| #15 | 假設驗證錯誤但 PM 抵抗 | `生成風險與回退計畫：`<br>`假設：[假設內容]`<br>`驗證結果：[具體數字或事實]`<br>`PM 反應：[PM 說了什麼]`<br>`專案：{專案名稱}`<br><br>範例：<br>`假設：員工願意從 email 換到新系統`<br>`驗證結果：問了 10 個員工，3 個說不想換`<br>`PM 反應：「他們之後會習慣的，繼續照計畫做」`<br>`專案：員工請假排班系統` | 產出量化風險表 + 降低風險的最小可行方案，記進 `log.md` 標注「PM 知情後決定繼續推進」 | ✅ |
| #16 | 專案中途大改方向 | `評估變更影響：[變更內容]，專案是 {專案名稱}` | 重寫「一句話說清楚」，逐一審查假設（✅/❌/⚠️），產出需要重新 Kickoff 的清單 | ✅ |
| #17 | 你不在時有人做決定 | `同步變更：[變更內容]，專案是 {專案名稱}` | 把變更摘要記進 `log.md`，標注「他人代決」；若影響核心假設立刻更新 `report.md`。建議補課順序：工程師→記錄→PO→更新地圖→主管 | ✅ |
| #18 | 同時負責多個專案、記錄混亂 | 預防：每條 log 加【專案名稱】標籤<br>補救：`幫我從以下 log 裡，抽出 {專案名稱} 的內容`（前提是有標籤） | 依標籤抽出特定專案條目，重建該專案的 `log.md`。若無標籤需手動整理。 | ⚠️ 補救部分待開發 |
| #19 | 一個 bug 影響多專案 | `這個 bug 同時影響 {專案A} 和 {專案B}，幫我對照兩個 report.md` | 分別讀取兩個 report.md，判斷是否同根本原因，產出影響摘要和建議處理順序 | ✅ |
| #20 | 假設被推翻，連鎖反應 | `追蹤影響鏈：[假設內容]，專案是 {專案名稱}` | 往上往下追假設依賴關係，列出受影響項目，跟 PM 說連鎖影響，立刻更新 `report.md` | ✅ |
| #21 | Kickoff 訪談後建立初版 | `建立初版：[貼上訪談筆記]，專案名稱是 {專案名稱}` | 產出 `kickoff.md`、`report.md` 與空白 `log.md` | ✅ |
| #22 | 週五更新邏輯地圖 | 單一專案：`週五到了`<br>多專案：`週五到了，更新 {專案名稱}` | 把 `log.md` 合入 `report.md`，標注「本週更新」，清空 `log.md`，提醒確認三件事 | ✅ |
| #23 | 新 PM onboarding | `新 PM 要加入了，幫我準備「{專案名稱}」的 onboarding 文件` | 用 `report.md` 當 onboarding 文件，產出關鍵待確認項與 30 分鐘議程，趁機重新量化成功定義 | ✅ |
| #24 | 邏輯地圖太大難維護 | `精簡邏輯地圖：{專案名稱}` | 按三個時機精簡（功能上線後/假設驗證後/方向大改後），舊內容移到「歷史記錄」不要刪 | ✅ |
| #25 | 資料 schema 頻繁變動 | `schema 異動：[描述]，專案是 {專案名稱}` | 在 `report.md` 核心假設清單標注「格式可能不穩定」，記進 `log.md` 追蹤 | ✅ |
| #26 | 被問「為什麼沒人早點發現」 | `被問為什麼沒人發現：[問題背景]，專案是 {專案名稱}` | 產出回應草稿：承認問題 → 說明是系統性原因（非個人疏失）→ 轉向解法和預防機制 | ✅ |
| #27 | 向主管報告（往上溝通） | `幫我準備向主管報告：{專案名稱}，重點是 [什麼]` | 從 `report.md` 擷取重點，用 Bottom line / 關鍵數字 / 風險 / 建議 的格式產出報告草稿 | ✅ |
| #28 | 收到突發資訊，還沒想清楚 | `突發狀況：[描述]，專案是 {專案名稱}` | 走一遍六個問題：知道什麼 / 不知道什麼 / 該問誰 / 最重要的一件事 / 最壞情況 / 第一個動作 | ✅ |
| #29 | 進度落後（時程變動） | `進度落後：[什麼卡住了]，專案是 {專案名稱}` | 讀 report.md 的時程和依賴，評估連帶影響哪些 Phase，更新時程，產出給主管的 heads-up（Bottom line 格式） | ✅ |
| #30 | 跨團隊依賴被卡住（資源變動） | `依賴卡住：[哪個依賴]，專案是 {專案名稱}` | 讀 report.md 找出這個依賴影響哪些功能/假設，判斷是否進關鍵路徑，建議是否升級處理，記進 log.md | ✅ |
| #31 | 需求一直被追加（範圍變動） | `這個需求算不算加太多：[新需求]，專案是 {專案名稱}` | 讀 kickoff.md 的「不做的是」邊界，判斷新需求是否越界，量化影響，產出跟 PM 談的說法（延後或換掉別的） | ✅ |

## 待開發功能說明

目前只剩一個部分尚未完整支援：

**#18 多專案 log 補救（沒有標籤時）：**
```
如果 log.md 裡的記錄沒有加【專案名稱】標籤，
Claude 無法自動判斷哪條屬於哪個專案。

遇到這個情況：
→ 需要手動整理 log.md，加上標籤後再抽取

預防方法（強烈建議）：
→ 每條 log 都加【專案名稱】標籤
→ 養成習慣後就不會遇到這個問題
```

**什麼時候補開發：**
```
你真實在工作中遇到 log 混亂補救的情境
→ 告訴我「遇到 #18 了」
→ 我幫你把這個情境加進 skill
```
