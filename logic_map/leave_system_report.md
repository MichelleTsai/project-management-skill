# 員工請假排班系統｜邏輯地圖

## 一句話說清楚這個專案
幫公司 HR 把手動用 email + Excel 處理請假的流程數位化，讓每筆請假從 15-20 分鐘縮短到 3 分鐘以內，並在公司擴編前準備好。

---

## 為什麼要做
- 如果不做，會發生什麼事：HR 每月花 25-30 小時手動處理，排班衝突持續發生；下半年擴編 50 人後問題會更嚴重
- Business Value：HR 處理每筆請假時間從 15-20 分鐘降到 3 分鐘，每月節省約 20-25 小時
- 量測指標：主要——HR 處理時間；次要——員工滿意度 >80% 正面評價、排班衝突事件歸零
- Justification：85% 員工問卷反映流程不方便 + HR 主管確認為最高優先痛點 + 擴編時間壓力
- Stakeholder 最在意的是：流程直覺不需要教育訓練，手機也能操作

## 功能邏輯

### Phase 1：請假申請和審核流程
- 在做什麼：員工線上申請請假，主管線上審核，HR 可查看所有人的請假狀態
- 為什麼這樣設計（不是直接做 App）：先做 web 版本降低開發複雜度，確認採用率後再考慮 App【本週更新：Jessica 在 Refine 正式確認 Phase 1 只做 web 版】
- 有沒有考慮過其他方式：Google Form（三年前試過，無審核流程，棄用）；需求文件已找到，放在 Confluence HR 專案資料夾【本週更新】
- 依賴關係：IT team（Okta SSO，預估 2-3 週）、HR Amy（假別規則清單，下週三）、法務 Kevin（特殊假別 checklist）【本週更新】
- 核心假設：員工願意換系統、SSO 整合可行

### Phase 2：HR 後台報表和統計
- 在做什麼：HR 可以看到各部門請假統計、剩餘假別、排班衝突預警
- 為什麼 Phase 2 才做：先確保 Phase 1 採用率穩定，再投資後台功能
- 依賴關係：Phase 1 資料累積
- 核心假設：Phase 1 成功採用

### Phase 3：與薪資系統串接（後續）
- 在做什麼：請假記錄自動同步薪資系統，減少 HR 二次輸入
- 為什麼 Phase 3：整合複雜度高，先確認核心流程穩定再做
- 核心假設：薪資系統有開放 API 或可整合
