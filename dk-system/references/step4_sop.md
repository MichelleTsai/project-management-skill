## Schema 驗證（新增）

在對 `kickoff.md` 或 sample data 進行自動產出 report 之前，請務必先做 schema 驗證。驗證步驟（通用）：

1. 取得 sample data（CSV 或 JSON）以及一份「預期欄位 schema」（JSON 格式，為欄位名稱清單或包含 fields 的物件）。
2. 使用 `tools/validate_schema.py --schema schema.json --sample sample.json`（或 CSV）執行檢查。
3. 若有差異（缺欄或多餘欄位），產生差異報告並標註為「待確認」，不要直接把未驗證的 sample 送入生產解析流程。

## 每週更新規則

根據 log.md 更新 report.md 時：
- 有變動的內容，在該行後面加上「本週更新」
- 新增的假設或風險，直接加在對應清單裡
- 已解決的問題，改為已確認並記錄答案
- 沒有變動的部分保持原樣
- 更新完後在最上面寫「上次更新：日期」
- 清空 log.md，只留標題和下週週次

---

## 不用等週五，立刻更新 report.md 的時機

- 專案大改方向
- 重要假設被推翻
- 主管突然問你，打開地圖發現過期了

**大改方向時的系統性審查：**
1. 重寫「一句話說清楚這個專案」
2. 逐一審查每個假設：✅ 還成立 / ❌ 已被推翻 / ⚠️ 需要重新驗證
3. 審查每個功能的設計邏輯是否還成立
4. 找 PM/主管重新確認成功定義和 timeline

---

## 週五確認三件事

更新完後，提醒 Michelle 確認：
1. 有沒有「待確認」欄位應該更新了？
2. 假設的驗證狀態有沒有更新對？
3. 還沒釐清的問題有沒有可以劃掉的？
