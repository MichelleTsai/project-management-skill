# Step 2：每週 Refine 會議框架

## 會議前（5 分鐘）
打開 report.md，確認：
- 上週有沒有未解決的問題？
- 有沒有假設需要被驗證？
- 這週最想搞清楚什麼？

---

## 會議中（插話時機）

### 正常情況：PO 說「決定 XXX」時進場
→ 「假設是 XXX，對嗎？」
→ 「假設錯了影響哪裡？」
→ 「會影響其他功能嗎？」

### 不知道假設是什麼時
→ 「這個決定背後最重要的考量是什麼？」

### 討論太快跟不上時
→ 「我確認一下——影響是 XXX，對嗎？」

### 發現假設未驗證時
→ 「這個假設我們有沒有驗證過？」
→ 如果沒有：「我們要怎麼驗證它？可以排進下週的行動項目嗎？」
→ 把驗證行動記進 log.md

### PO 和工程師意見不一致時
TPM 的角色：把假設說清楚、把風險量化、讓 PO 做最終決定
→ 先問工程師：「如果這樣做，最壞的情況是什麼？影響多少人？」
→ 再翻譯給 PO：「X% 的使用者可能會 YYY，這個風險你可以接受嗎？」

### 完全聽不懂技術討論時
方式一（最推薦）：「等一下，這個決定對使用者的影響是什麼？技術細節先跳過，但我需要知道結果。」
方式二：先記下關鍵詞，會議後問
方式三：會議後請工程師補課

### 工程師說「做不到」時
→ Step 1：「你說做不到，是技術上不可能，還是成本太高？」
→ Step 2：「你需要這個功能，最重要的原因是什麼？」（問 PM）
→ Step 3：找中間解法，滿足真正需求的最低成本方案

---

## 會議後（10 分鐘）

記進 log.md：
```
【Refine Week X】
決定：
原因（假設）：
影響範圍：
還沒解決的問題：
待決定：XXX（誰負責？預計回覆時間：XXX）
```

**決定在會議中沒有被解決時：**
→ 記進 log.md：「待決定：XXX，由誰確認，預計回覆時間：XXX」
→ 主動追蹤，下次 refine 前確認有沒有答案
