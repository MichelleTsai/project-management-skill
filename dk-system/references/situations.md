# 24 個情境處理指南

## 進入新專案

**#01 全新專案，你是第一個 TPM**
→ 用 step1_kickoff.md 引導 Kickoff → 建立三個檔案

**#02 全新專案，PM 答案很模糊**
→ Step 1 + 具體化技巧（舉例法/對比法/量化法）
→ 三個都試了還是模糊：「我現在理解的是 XXX，你覺得這樣說對嗎？」
→ 把待確認的記進 report.md，下次 refine 確認

**#03 接手進行中的專案，有舊記錄**
→ 先自己讀舊記錄（不要讓別人幫你總結）
→ 用補課路徑確認哪些假設還成立
→ 建立或更新 report.md
⚠️ 舊記錄品質差時，補課深度要加重

**#04 前任 TPM 突然離職，什麼都沒留**
→ 補課順序：工程師 → 會議記錄 → PO → 更新 report.md → 主管
→ 用 report.md 當 onboarding 文件給新 PM
→ 趁新 PM 上任重新量化成功定義

## 每週 Refine 會議

**#05 正常的 Refine，沒有特別狀況**
→ 照 step2_refine.md 的會議前/中/後框架走

**#06 Refine 中發現假設從來沒被驗證**
→ 「這個假設我們有沒有驗證過？」
→ 排進下週行動項目，記進 log.md

**#07 PO 和工程師意見不一致**
→ 先問工程師最壞情況，再翻譯給 PO，讓 PO 做決定
→ TPM 不做決定，只幫兩邊把資訊說清楚

**#08 討論內容你完全聽不懂**
→ 方式一：「這個決定對使用者的影響是什麼？」
→ 方式二：先記關鍵詞，會議後問
→ 方式三：會議後請工程師用非技術語言補課

**#09 決定在會議中沒有被解決**
→ 記進 log.md：「待決定：XXX，誰負責，預計回覆時間：XXX」
→ 主動追蹤，下次 refine 前確認

**#10 工程師說「這個需求做不到」**
→ 澄清「做不到」的意思（技術不可能 vs 成本太高）
→ 澄清需求背後的真正目的
→ 找中間解法，滿足需求的最低成本方案

## 遇到問題 / 主管問你

**#11 主管問你一個你知道答案的問題**
→ 打開 report.md → 確認假設 → 第一層追問 → 跟主管說影響

**#12 邏輯地圖有，但標注「待確認」**
→ 不要猜，告訴主管：「我現在不知道，但我可以在 X 小時內給你答案。你需要答案的時間是什麼時候？」

**#13 邏輯地圖完全沒有這個資訊**
→ 一句話說清楚背景，三分鐘跨部門確認
→ 「我負責 XXX 專案，想確認 ZZZ，你有三分鐘嗎？」

**#14 工程師一直給技術答案**
→ 第一層追問 → 第二層追問 → 最後一招
→ 最後一招：「我需要的是對 timeline 和品質的影響，不是技術原因」

**#15 假設驗證是錯的，但 PM 不接受**
→ 先理解 PM 的考量 → 量化風險 → 提出降低風險的最小可行方案
→ 決定後記進 log.md：「PM 知情後決定繼續推進」

## 突發狀況

**#16 專案中途被要求大改方向**
→ 重寫一句話說清楚 → 逐一審查假設 → 審查功能邏輯 → 重新 Kickoff
⚠️ 改變幅度決定哪些 Step 需要完整重做

**#17 你不在的時候有人做了重大決定**
→ 補課順序：工程師 → 會議記錄 → PO → 更新 report.md → 主管
→ 特別標注「你不在時決定的」
❌ 不要一回來就直接找主管問（問題太模糊）

**#18 同時負責多個專案，記錄混亂**
→ 每個專案獨立的三個檔案
→ log.md 每條都要加專案標籤：「【HBM3E】週一 - Kevin 回報...」
→ 已經混了：請 Claude 從 log 裡抽出特定專案的內容

**#19 一個 bug 同時影響兩個專案**
→ 分別打開兩個 report.md
→ 判斷是否同根本原因（是 → 一起解決；否 → 按嚴重性排序）
→ 跟主管說清楚兩個專案的連動關係

**#20 一個假設被推翻，連鎖反應**
→ 往上往下追假設依賴關係
→ 跟 PM 說連鎖影響
→ 立刻更新 report.md（不等週五）

## 資料與採用風險情境（新增）

**#25 資料 schema 頻繁變動**
→ 影響：自動化解析器失敗或輸出欄位不一致
→ 作法：先執行 schema 驗證步驟（step4_sop.md 指定），若驗證失敗標註為「待確認」並通知資料擁有團隊

**#26 PO/PM 回覆延遲或不完整**
→ 影響：Kickoff 資訊不足，會議需延長或增加一次追問
→ 作法：先收集最小必答（必備欄位）並在會議中追問剩餘項目，會後把缺項標註並追蹤

**#27 採用率低（工程師拒用或不習慣）**
→ 影響：自動化投資回收慢或失敗
→ 作法：設定採用率門檻與回退流程（見 SKILL.md 的採用與回退策略），安排示範、early-adopter 支援與快速修正

**#28 遺留系統可重用性不確定**
→ 影響：需要額外開發或資料轉換
→ 作法：先評估遺留系統的介面與資料格式，判斷可重用度並納入 PoC 範圍或排除

## 邏輯地圖維護

**#21 Kickoff 訪談後建立邏輯地圖初版**
→ 訪談記錄存進 kickoff.md → 建立初版 Prompt → 存進 report.md → 建立空白 log.md

**#22 週五用流水帳更新邏輯地圖**
→ 讀取 report.md + log.md → 更新 report.md → 清空 log.md → 確認三件事

**#23 新 PM 加入，用邏輯地圖幫她了解專案**
→ report.md 當 onboarding 文件
→ 30 分鐘對齊，特別確認「待確認」欄位
→ 趁機重新量化成功定義

**#24 邏輯地圖越來越大，難以維護**
→ 基準：讓新加入的人 15 分鐘內了解現在最重要的事
→ 精簡時機：功能上線後 / 假設驗證後 / 方向大改後
→ 舊內容移到「歷史記錄」區塊，不要刪
→ 請 Claude 幫你判斷哪些可以移到歷史記錄
