# 28 個情境處理指南

## 進入新專案

**#01 全新專案，你是第一個 TPM**
→ 用 step1_kickoff.md 引導 Kickoff → 建立三個檔案

**#02 全新專案，PM 答案很模糊**
→ Step 1 + 具體化技巧（舉例法/對比法/量化法）
→ 三個都試了還是模糊：「我現在理解的是 XXX，你覺得這樣說對嗎？」
→ 把待確認的記進 report.md，下次 refine 確認

**#03 接手進行中的專案，有舊記錄**
→ 先自己讀舊記錄（不要讓別人幫你總結）
→ 用補課路徑確認哪些假設還成立
→ 建立或更新 report.md
⚠️ 舊記錄品質差時，補課深度要加重

如何使用 skill：
- 指令：`整合舊記錄：[貼上舊記錄內容]，專案名稱是 {專案名稱}`
- Skill 會解析舊記錄整理成 kickoff.md，產出 report.md 初版，標注「由舊記錄重建，假設需重新驗證」

**#04 前任 TPM 突然離職，什麼都沒留**
→ 先問三個最小問題：專案叫什麼 / PM 是誰 / 最緊急的是什麼
→ 補課順序：工程師 → 會議記錄 → PO → 更新 report.md → 主管
→ 用 report.md 當 onboarding 文件給新 PM

如何使用 skill：
- 指令：`重建 Kickoff`
- Skill 會先問三個最小問題，建立最小版 kickoff.md（其餘欄位標「待補課」），列出補課順序

## 每天記錄 / 每週 Refine 會議

**#05 每天記錄 / 正常的 Refine**
→ 照 step2_refine.md 的會議前/中/後框架走
→ 每天發生的事（不只 Refine）都記進 log.md

如何使用 skill：
- 指令：`幫我記進 log：[貼上今天發生的事或 Refine 重點]，專案名稱是 {專案名稱}`
- Skill 會加【專案名稱】標籤後記進 log.md，週五統一更新 report.md

**#06 Refine 中發現假設從來沒被驗證**
→ 「這個假設我們有沒有驗證過？」
→ 排進下週行動項目，記進 log.md

如何使用 skill：
- 指令：`標註需驗證：[假設內容]，專案是 {專案名稱}，誰負責驗證`
- Skill 會在 log.md 加入驗證任務，同時在 report.md 核心假設清單標注 ⚠️ 未驗證

**#07 PO 和工程師意見不一致**
→ 先問工程師最壞情況，再翻譯給 PO，讓 PO 做決定
→ TPM 不做決定，只幫兩邊把資訊說清楚

如何使用 skill：
- 指令：`生成討論摘要：[PO 說的] vs [工程師說的]，幫我整理決策選項，專案是 {專案名稱}`
- Skill 會輸出決策選項比較表（選項/影響/風險/建議），並在 log.md 建立待決事項

**#08 討論內容你完全聽不懂**
→ 方式一：「這個決定對使用者的影響是什麼？」
→ 方式二：先記關鍵詞，會議後問
→ 方式三：會議後請工程師用非技術語言補課

如何使用 skill：
- 指令：`解釋這段`，並貼上技術摘錄
- Skill 會回寫為非技術化摘要，並列出 2–3 個會後追問範本

**#09 決定在會議中沒有被解決**
→ 記進 log.md：「待決定：XXX，誰負責，預計回覆時間：XXX」
→ 主動追蹤，下次 refine 前確認

如何使用 skill：
- 指令：`標註待決事項：[什麼事情]，由 [誰] 負責，預計 [什麼時間] 回覆，專案是 {專案名稱}`
- Skill 會加入 log.md，下次 Refine 前追蹤

**#10 工程師說「這個需求做不到」**
→ 澄清「做不到」的意思（技術不可能 vs 成本太高）
→ 澄清需求背後的真正目的
→ 找中間解法，滿足需求的最低成本方案

如何使用 skill：
- 指令：`評估替代方案：[工程師說的理由]，專案是 {專案名稱}`
- Skill 會生成技術限制摘要與 2–3 個替代方案（含預估時間/風險）

## 遇到問題 / 主管問你

**#11 主管問你一個你知道答案的問題**
→ 打開 report.md → 確認假設 → 第一層追問 → 跟主管說影響

如何使用 skill：
- 指令：`快速回覆主管：[問題內容]，專案是 {專案名稱}`
- Skill 會從 report.md 擷取要點，整理為「問題 + 解法 + 需要主管決定什麼」

**#12 邏輯地圖有，但標注「待確認」**
→ 不要猜，告訴主管：「我現在不知道，但我可以在 X 小時內給你答案。你需要答案的時間是什麼時候？」

如何使用 skill：
- 指令：`我不知道這個答案，幫我看 {專案名稱} 的 report.md，有沒有相關的待確認項`
- Skill 會整理出相關的待確認條目並建議責任人與期限

**#13 邏輯地圖完全沒有這個資訊**
→ 一句話說清楚背景，三分鐘跨部門確認
→ 「我負責 XXX 專案，想確認 ZZZ，你有三分鐘嗎？」

如何使用 skill：
- 指令：`補資料：[問題內容]，專案是 {專案名稱}`
- Skill 會生成追問清單或跨部門確認的一句話說法

**#14 工程師一直給技術答案**
→ 第一層追問 → 第二層追問 → 最後一招
→ 最後一招：「我需要的是對 timeline 和品質的影響，不是技術原因」

如何使用 skill：
- 指令：`轉成決策選項：[貼技術內容]，專案是 {專案名稱}`
- Skill 會輸出 PM 可決策的選項並標示每個選項對時程/成本/品質的影響

**#15 假設驗證是錯的，但 PM 不接受**
→ 先理解 PM 的考量 → 量化風險 → 提出降低風險的最小可行方案
→ 決定後記進 log.md：「PM 知情後決定繼續推進」

如何使用 skill：
- 指令：`生成風險與回退計畫：假設[...] / 驗證結果[...] / PM 反應[...] / 專案：{專案名稱}`
- Skill 會產出量化風險表、降低風險的最小可行方案，以及可以拿去跟 PM 談的說法

## 突發狀況

**#16 專案中途被要求大改方向**
→ 重寫一句話說清楚 → 逐一審查假設 → 審查功能邏輯 → 重新 Kickoff
⚠️ 改變幅度決定哪些 Step 需要完整重做

如何使用 skill：
- 指令：`評估變更影響：[變更內容]，專案是 {專案名稱}`
- Skill 會重寫「一句話說清楚」，逐一審查假設（✅/❌/⚠️），產出需要重新 Kickoff 的清單

**#17 你不在的時候有人做了重大決定**
→ 補課順序：工程師 → 會議記錄 → PO → 更新 report.md → 主管
→ 特別標注「你不在時決定的」
❌ 不要一回來就直接找主管問（問題太模糊）

如何使用 skill：
- 指令：`同步變更：[變更內容]，專案是 {專案名稱}`
- Skill 會把變更摘要記進 log.md 標「他人代決」；若影響核心假設立刻更新 report.md

**#18 同時負責多個專案，記錄混亂**
→ 每個專案獨立的三個檔案
→ log.md 每條都要加專案標籤：「【員工請假排班系統】週一 - Kevin 回報...」
→ 已經混了：請 skill 從 log 中抽出特定專案的內容

如何使用 skill：
- 預防：每條 log 加【專案名稱】標籤
- 補救：`幫我從以下 log 裡，抽出 {專案名稱} 的內容`（前提是有標籤）
- ⚠️ 若無標籤需手動整理後才能抽取

**#19 一個 bug 同時影響兩個專案**
→ 分別打開兩個 report.md
→ 判斷是否同根本原因（是 → 一起解決；否 → 按嚴重性排序）
→ 跟主管說清楚兩個專案的連動關係

如何使用 skill：
- 指令：`這個 bug 同時影響 {專案A} 和 {專案B}，幫我對照兩個 report.md`
- Skill 會分別讀取兩個 report.md，判斷是否同根本原因，產出影響摘要和建議處理順序

**#20 一個假設被推翻，連鎖反應**
→ 往上往下追假設依賴關係
→ 跟 PM 說連鎖影響
→ 立刻更新 report.md（不等週五）

如何使用 skill：
- 指令：`追蹤影響鏈：[假設內容]，專案是 {專案名稱}`
- Skill 會向上/下追溯依賴，列出受影響項目並產出優先修正清單

## 邏輯地圖維護

**#21 Kickoff 訪談後建立邏輯地圖初版**
→ 訪談記錄存進 kickoff.md → 建立初版 Prompt → 存進 report.md → 建立空白 log.md

如何使用 skill：
- 指令：`建立初版：[貼上訪談筆記]，專案名稱是 {專案名稱}`

**#22 週五用流水帳更新邏輯地圖**
→ 讀取 report.md + log.md → 更新 report.md → 清空 log.md → 確認三件事

如何使用 skill：
- 單一專案：`週五到了`
- 多專案：`週五到了，更新 {專案名稱}`

**#23 新 PM 加入，用邏輯地圖幫她了解專案**
→ report.md 當 onboarding 文件
→ 30 分鐘對齊，特別確認「待確認」欄位
→ 趁機重新量化成功定義

如何使用 skill：
- 指令：`新 PM 要加入了，幫我準備「{專案名稱}」的 onboarding 文件`
- Skill 會產出精簡 onboarding 摘要與 30 分鐘對齊議程，標示首要待確認項目

**#24 邏輯地圖越來越大，難以維護**
→ 基準：讓新加入的人 15 分鐘內了解現在最重要的事
→ 精簡時機：功能上線後 / 假設驗證後 / 方向大改後
→ 舊內容移到「歷史記錄」區塊，不要刪

如何使用 skill：
- 指令：`精簡邏輯地圖：{專案名稱}`
- Skill 會分析 report.md，把可移到歷史記錄的段落標註出來

**#25 資料 schema 頻繁變動**
→ TPM 的角色是「標注這個假設有風險」，不是執行驗證
→ 在 report.md 核心假設清單標注「格式可能不穩定」

如何使用 skill：
- 指令：`schema 異動：[描述]，專案是 {專案名稱}`
- Skill 會在 report.md 標注風險，記進 log.md 追蹤

## 溝通與突發應對

**#26 被問「為什麼沒人早點發現」**
這是政治陷阱型問題，重點不是甩鍋也不是攬全責，而是把焦點導向解法。
→ 第一步：承認問題確實存在（不辯解、不甩鍋）
→ 第二步：說明這是系統性原因（流程/資訊落差），不是某個人的疏失
→ 第三步：轉向解法——現在要怎麼修，以及未來怎麼預防

如何使用 skill：
- 指令：`被問為什麼沒人發現：[問題背景]，專案是 {專案名稱}`
- Skill 會產出回應草稿：承認 → 系統性原因 → 轉向解法和預防機制

**#27 向主管報告（往上溝通）**
主管沒時間看細節，要先講結論。
→ 從 report.md 擷取重點
→ 用固定格式：Bottom line（結論先講）/ 關鍵數字 / 風險 / 建議

如何使用 skill：
- 指令：`幫我準備向主管報告：{專案名稱}，重點是 [什麼]`
- Skill 會從 report.md 擷取重點，用 Bottom line / 關鍵數字 / 風險 / 建議 的格式產出報告草稿

**#28 收到突發資訊時，先想清楚再行動**
收到新資訊或突發狀況時，先回答這六個問題，不要急著回覆或做決定。

1. 先停下來｜給自己 2 分鐘，問自己：我現在知道什麼？不知道什麼？
2. 把資訊分類｜已確認的事實是什麼？還不確定的是什麼？誰知道我不知道的事？
3. 評估影響範圍｜這件事影響誰？多久？最壞情況？先想好 B 計畫
4. 決定誰來做什麼｜哪些事只有我能做？哪些可以讓別人做？
5. 決定溝通順序｜誰需要第一個知道？用什麼方式說？Bottom line 先說
6. 設定 Checkpoint｜什麼時候重新檢查？A 計畫不順時什麼時候啟動 B 計畫？

如何使用 skill：
- 指令：`突發狀況：[描述]，專案是 {專案名稱}`
- Skill 會帶你走一遍上面六個問題，把突發資訊記進 log.md，若影響核心假設提醒是否立刻更新 report.md
