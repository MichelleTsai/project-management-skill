# 28 個情境處理指南

## 進入新專案

**#01 全新專案，你是第一個 TPM**
→ 用 step1_kickoff.md 引導 Kickoff → 建立三個檔案

**#02 全新專案，PM 答案很模糊**
→ Step 1 + 具體化技巧（舉例法/對比法/量化法）
→ 三個都試了還是模糊：「我現在理解的是 XXX，你覺得這樣說對嗎？」
→ 把待確認的記進 report.md，下次 refine 確認

**#03 接手進行中的專案，有舊記錄**
→ 先自己讀舊記錄（不要讓別人幫你總結）
→ 用補課路徑確認哪些假設還成立
→ 建立或更新 report.md
⚠️ 舊記錄品質差時，補課深度要加重

如何使用 skill（一步到位）：
- 指令/口述範例：`整合舊記錄` 或 「請幫我把這些舊會議記錄/報表整合」，並上傳檔案或貼內容。
- Skill 會解析並匯入 `logic_map/{project}_log.md`，在 `report.md` 新增「舊記錄摘要（可信度）」並標注需補課假設。


**#04 前任 TPM 突然離職，什麼都沒留**
→ 補課順序：工程師 → 會議記錄 → PO → 更新 report.md → 主管
→ 用 report.md 當 onboarding 文件給新 PM
→ 趁新 PM 上任重新量化成功定義

如何使用 skill（一步到位）：
- 指令/口述範例：`重建 Kickoff` 或 「前任離職，請幫我快速重建 Kickoff 問卷」。
- Skill 會發送最小必答問卷，生成最小版 `logic_map/{project}_kickoff.md`，並輸出 onboarding checklist（誰知道什麼、下一步負責人）。

## 每週 Refine 會議

**#05 正常的 Refine，沒有特別狀況**
→ 照 step2_refine.md 的會議前/中/後框架走

**#06 Refine 中發現假設從來沒被驗證**
→ 「這個假設我們有沒有驗證過？」
→ 排進下週行動項目，記進 log.md

**#07 PO 和工程師意見不一致**
→ 先問工程師最壞情況，再翻譯給 PO，讓 PO 做決定
→ TPM 不做決定，只幫兩邊把資訊說清楚

如何使用 skill（一步到位）：
- 指令/口述範例：`生成討論摘要` 或 貼上會議摘錄並說「整理分歧與決策選項」。
- Skill 會輸出「決策選項比較表」(選項/影響/風險/建議)，並在 `log.md` 建立待決事項與建議責任人。


**#08 討論內容你完全聽不懂**
→ 方式一：「這個決定對使用者的影響是什麼？」
→ 方式二：先記關鍵詞，會議後問
→ 方式三：會議後請工程師用非技術語言補課

如何使用 skill（一步到位）：
- 指令/口述範例：`解釋這段` 並貼上技術摘錄，skill 會回寫為非技術化摘要，並列出 2–3 個會後追問範本。


**#09 決定在會議中沒有被解決**
→ 記進 log.md：「待決定：XXX，誰負責，預計回覆時間：XXX」
→ 主動追蹤，下次 refine 前確認

如何使用 skill（一步到位）：
- 指令/口述範例：會議結束時說「標註待決事項」，或把未決項貼上，skill 會加入 `log.md` 並在下次週報置頂提醒追蹤。


**#10 工程師說「這個需求做不到」**
→ 澄清「做不到」的意思（技術不可能 vs 成本太高）
→ 澄清需求背後的真正目的
→ 找中間解法，滿足需求的最低成本方案

如何使用 skill（一步到位）：
- 指令/口述範例：`評估替代方案`，並貼上工程師的理由，skill 會生成技術限制摘要與 2–3 個替代方案（含預估時間/風險）。

## 遇到問題 / 主管問你

**#11 主管問你一個你知道答案的問題**
→ 打開 report.md → 確認假設 → 第一層追問 → 跟主管說影響

如何使用 skill（一步到位）：
- 指令/口述範例：`快速回覆主管：<問題>`，skill 會從 `report.md`/`log.md` 擷取要點並回傳 2–3 句簡短回覆，並附上參考段落連結。


**#12 邏輯地圖有，但標注「待確認」**
→ 不要猜，告訴主管：「我現在不知道，但我可以在 X 小時內給你答案。你需要答案的時間是什麼時候？」

如何使用 skill（一步到位）：
- 指令/口述範例：`列出待確認項` 或 `待確認狀態`，skill 會整理出所有標注為「待確認」的條目並建議責任人與期限。

**#13 邏輯地圖完全沒有這個資訊**
→ 一句話說清楚背景，三分鐘跨部門確認
→ 「我負責 XXX 專案，想確認 ZZZ，你有三分鐘嗎？」

如何使用 skill（一步到位）：
- 指令/口述範例：`補資料：<問題>` 或 `請幫我補齊這個資訊`，skill 會生成追問清單或問卷，協助取得缺失資訊。

**#14 工程師一直給技術答案**
→ 第一層追問 → 第二層追問 → 最後一招
→ 最後一招：「我需要的是對 timeline 和品質的影響，不是技術原因」

如何使用 skill（一步到位）：
- 指令/口述範例：`轉成決策選項` 並貼上技術回覆，skill 會輸出對 PM 可決策的選項並標示每個選項對時程/成本/品質的影響。


**#15 假設驗證是錯的，但 PM 不接受**
→ 先理解 PM 的考量 → 量化風險 → 提出降低風險的最小可行方案
→ 決定後記進 log.md：「PM 知情後決定繼續推進」

如何使用 skill（一步到位）：
- 指令/口述範例：`生成風險與回退計畫`，skill 會產出量化風險表、回退門檻與可操作步驟，並在 `report.md` 中標註負責人。

## 突發狀況

**#16 專案中途被要求大改方向**
→ 重寫一句話說清楚 → 逐一審查假設 → 審查功能邏輯 → 重新 Kickoff
⚠️ 改變幅度決定哪些 Step 需要完整重做

如何使用 skill（一步到位）：
- 指令/口述範例：`評估變更影響` 並貼上變更摘要，skill 會產出變更影響快照（需重做的 Step、估計工時、優先順序），並生成新的 Kickoff 草案。


**#17 你不在的時候有人做了重大決定**
→ 補課順序：工程師 → 會議記錄 → PO → 更新 report.md → 主管
→ 特別標注「你不在時決定的」
❌ 不要一回來就直接找主管問（問題太模糊）

如何使用 skill（一步到位）：
- 指令/口述範例：`同步變更` 並貼上會議記錄或 email，skill 會把變更摘要加入 `report.md` 並標註「他人代決」，同時列出需補課的清單與負責人。


**#18 同時負責多個專案，記錄混亂**
→ 每個專案獨立的三個檔案
→ log.md 每條都要加專案標籤：「【HBM3E】週一 - Kevin 回報...」
→ 已經混了：請系統從 log 中抽出特定專案的內容

如何使用 skill（一步到位）：
- 指令/口述範例：`整理我的專案清單` 或 `抽出專案 {project}`，skill 會依專案標籤從混合的 log 提取條目，重建各專案的 `log.md`，並產出 `overview.md` 幫你快速切換上下文。


**#19 一個 bug 同時影響兩個專案**
→ 分別打開兩個 report.md
→ 判斷是否同根本原因（是 → 一起解決；否 → 按嚴重性排序）
→ 跟主管說清楚兩個專案的連動關係

如何使用 skill（一步到位）：
- 指令/口述範例：`cross-project bug: <描述>`，skill 會生成影響矩陣、跨專案通知範本，並在受影響的 `report.md` 中記錄影響與優先順序。


**#20 一個假設被推翻，連鎖反應**
→ 往上往下追假設依賴關係
→ 跟 PM 說連鎖影響
→ 立刻更新 report.md（不等週五）

如何使用 skill（一步到位）：
- 指令/口述範例：`追蹤影響鏈 <假設ID或內容>`，skill 會向上/下追溯依賴，列出受影響項目並產出優先修正清單與建議緩解措施。

## 資料與採用風險情境（新增）

**#25 資料 schema 頻繁變動**
→ 影響：自動化解析器失敗或輸出欄位不一致
→ 作法：先執行 schema 驗證步驟（step4_sop.md 指定），若驗證失敗標註為「待確認」並通知資料擁有團隊

**#26 PO/PM 回覆延遲或不完整**
→ 影響：Kickoff 資訊不足，會議需延長或增加一次追問
→ 作法：先收集最小必答（必備欄位）並在會議中追問剩餘項目，會後把缺項標註並追蹤

**#27 採用率低（工程師拒用或不習慣）**
→ 影響：自動化投資回收慢或失敗
→ 作法：設定採用率門檻與回退流程（見 SKILL.md 的採用與回退策略），安排示範、early-adopter 支援與快速修正

**#28 遺留系統可重用性不確定**
→ 影響：需要額外開發或資料轉換
→ 作法：先評估遺留系統的介面與資料格式，判斷可重用度並納入 PoC 範圍或排除

## 邏輯地圖維護

**#21 Kickoff 訪談後建立邏輯地圖初版**
→ 訪談記錄存進 kickoff.md → 建立初版 Prompt → 存進 report.md → 建立空白 log.md

**#22 週五用流水帳更新邏輯地圖**
→ 讀取 report.md + log.md → 更新 report.md → 清空 log.md → 確認三件事

**#23 新 PM 加入，用邏輯地圖幫她了解專案**
→ report.md 當 onboarding 文件
→ 30 分鐘對齊，特別確認「待確認」欄位
→ 趁機重新量化成功定義

如何使用 skill（一步到位）：
- 指令/口述範例：`匯出導覽版 {project}`，skill 會產出精簡 onboarding 摘要與 30 分鐘對齊議程，並標示首要待確認項目。


**#24 邏輯地圖越來越大，難以維護**
→ 基準：讓新加入的人 15 分鐘內了解現在最重要的事
→ 精簡時機：功能上線後 / 假設驗證後 / 方向大改後
→ 舊內容移到「歷史記錄」區塊，不要刪
→ 請系統幫你判斷哪些可以移到歷史記錄

如何使用 skill（一步到位）：
- 指令/口述範例：`精簡邏輯地圖 {project}`，skill 會分析 `report.md`，建議分割策略（功能或時間窗）、把可移到歷史記錄的段落標註並產出維護 SOP 範本。
